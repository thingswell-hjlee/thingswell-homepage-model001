# Admin panel with password management, dashboard, and session timeout

This change adds a full admin section to a React SPA backed by AWS Cognito: a password change page, a forgot-password/reset flow, an admin dashboard with content stats, a profile page, session timeout handling with a 5-minute warning banner, and a sidebar navigation component tying the admin pages together. The approach uses Cognito's public API directly (no SDK) for the new `forgotPassword` and `confirmForgotPassword` operations, matching the existing `signIn`/`changePassword` pattern. A 60-second polling interval in `AuthContext` monitors token expiry and force-signs-out when the token expires.

**Watch for:** The session timeout monitor and `getSession`'s token-refresh logic both trigger at the 5-minute-before-expiry boundary, creating a race where the warning banner may flash and disappear or the user gets signed out despite a valid refresh token (confirmed). The `/admin/og-settings` route remains unprotected by `ProtectedRoute` while all other admin routes are protected (confirmed). The Profile page duplicates the JWT-decoding logic already in `auth.js` (confirmed).

## High-level view

The password flows (change and forgot/reset) follow the project's existing `{ data, error }` Cognito API wrapper pattern. The forgot-password page is publicly accessible (correctly so), while the change-password page is behind `ProtectedRoute`.

The dashboard fetches counts from existing API functions on mount and renders stat cards, quick links, and recent posts. It does not use `AdminLayout`, so it renders without the sidebar — an inconsistency with the Profile page which wraps content in `AdminLayout`.

Session timeout monitoring polls `localStorage` every 60 seconds from `AuthContext`. The 5-minute warning threshold collides with `getSession`'s proactive token-refresh window (also 5 minutes before expiry), meaning any API call during the warning period will silently refresh the token and push the expiry forward, or if the user is idle, the monitor signs out without attempting a refresh.

The admin menu item ("관리자") is conditionally shown via a `requireAuth` flag filtered in the Menu component.

## Session timeout race with token refresh

`AuthContext.checkSessionTimeout` reads `thingswell_token_expiry` from localStorage and compares it to `Date.now()`. When `timeRemaining <= 0`, it calls `authSignOut()`. Meanwhile, `getSession` in `auth.js` already has logic to proactively refresh the token when the current time exceeds `expiresAt - fiveMinutes`:

```js
// auth.js line 270
if (expiresAt && now > expiresAt - fiveMinutes) {
  const refreshed = await refreshSession();
  return { data: { session: refreshed }, error: null };
}
```

The 5-minute warning constant in `AuthContext` (`EXPIRY_WARNING_MS = 5 * 60 * 1000`) exactly matches this window. Three scenarios emerge:

1. A dashboard stat fetch triggers `getSession` during the warning period — token refreshes, `TOKEN_EXPIRY` updates to an hour from now, warning banner disappears on next 60s poll. The user sees the banner briefly then it vanishes with no explanation.
2. The user is idle, no API calls happen, the token expires naturally — `checkSessionTimeout` fires `authSignOut()` without ever attempting a refresh. The refresh token may still be valid.
3. The `checkSessionTimeout` and an API-triggered `getSession` fire concurrently — `authSignOut` clears localStorage while `refreshSession` is mid-flight, potentially causing a failed refresh that also triggers `notifyListeners('SIGNED_OUT')`.

The fix is for the timeout handler to call `getSession()` (which attempts refresh) instead of raw localStorage reads, and only sign out if `getSession` returns a null session.

## Forgot-password flow exposure

The Cognito `ForgotPassword` API action is subject to rate limiting on the Cognito side but the page has no client-side rate limiting or CAPTCHA. Repeated requests with arbitrary emails will generate Cognito SMS/email charges if the user pool is configured with SMS-based delivery. For a small admin-only site this is low-severity, but worth noting. (likely)

## Inconsistent AdminLayout across admin pages

`Profile.jsx` wraps its content inside `<AdminLayout>` which renders the sidebar navigation. `Dashboard.jsx` and `ChangePassword.jsx` use only `<BaseLayout>`. The sidebar's navigation links point to Dashboard and ChangePassword, so clicking those links from the Profile page navigates away from the sidebar into a page without it. The user has no consistent way to get back except the browser back button or the header navigation.

This suggests AdminLayout was intended to wrap all admin routes (at the router level, like a nested route layout) rather than being applied per-page. The current state creates a disjointed navigation experience.

## Login redirect behavior change

The login page now redirects authenticated users to `/admin/dashboard` instead of `/` (home), with `?redirect=` query parameter support. This means any user who hits `/login` directly (bookmark, shared link) will land on the dashboard after login rather than the homepage — a behavior change for any non-admin users if the site ever has them.
